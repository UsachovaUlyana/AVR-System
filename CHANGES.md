# Журнал изменений проекта — Сайт ООО «АВР Систем»

Этот файл фиксирует все ключевые архитектурные и технические решения, принятые в ходе разработки.
Используется для актуализации содержания отчёта по практике.

---

## [2026-06-16] Инициализация проекта и выбор стека

**Что сделано:** Проект создан с нуля. Выбран технологический стек.

**Изменение относительно первоначального плана:**
- Первоначально в отчёте указано хранение данных в `localStorage` браузера.
- **Принято решение:** использовать PostgreSQL как полноценную реляционную СУБД.
- **Причина:** реальная система управления заявками требует надёжного персистентного хранилища, поддержки одновременной работы нескольких пользователей и возможности масштабирования.

**Итоговый стек:**
- Frontend: React 18 + Vite + React Router DOM + CSS
- Backend: Node.js + Express (REST API)
- База данных: PostgreSQL 16
- Инфраструктура: Docker + Docker Compose

## [2026-06-16] Контейнеризация с Docker

**Что сделано:** Весь проект обёрнут в Docker-контейнеры.

**Изменение относительно первоначального плана:**
- В отчёте не упоминается Docker.
- **Принято решение:** использовать Docker Compose с тремя отдельными сервисами.
- **Причина:** обеспечение воспроизводимой среды выполнения, независимость от локальной конфигурации, упрощение развёртывания на хостинге.

**Структура контейнеров:**
- `postgres` — контейнер с СУБД PostgreSQL 16, named volume `pgdata` для сохранности данных
- `backend` — контейнер с Node.js/Express API, зависит от `postgres` (healthcheck)
- `frontend` — контейнер с Vite dev-сервером, проксирует `/api` на `backend:3001`

**Файлы:** `docker-compose.yml`, `backend/Dockerfile`, `frontend/Dockerfile`

## [2026-06-16] Схема базы данных

**Что сделано:** Разработана реляционная схема из трёх таблиц.

**Изменение относительно первоначального плана:**
- В отчёте модель данных описана как единый объект (плоский JSON) с полем `statusHistory` как массивом внутри объекта.
- **Принято решение:** вынести историю статусов в отдельную таблицу `status_history` с внешним ключом.
- **Причина:** нормализация данных, корректное хранение временны́х меток каждого изменения, возможность сортировки и фильтрации истории на уровне SQL.

**Таблицы:** `users`, `orders`, `status_history`
**Файл:** `backend/migrations/init.sql`

## [2026-06-16] Аутентификация и сессии

**Что сделано:** Реализована серверная аутентификация через `express-session`.

**Изменение относительно первоначального плана:**
- В отчёте описана авторизация через хранение роли в `localStorage`.
- **Принято решение:** хранить сессию на сервере (`express-session`), передавать cookie клиенту.
- **Причина:** клиентское хранение роли небезопасно — пользователь может подменить значение в `localStorage` и получить доступ к чужим данным. Серверные сессии исключают эту угрозу.

**Файл:** `backend/routes/auth.js`, `backend/index.js`

## [2026-06-16] Публичная форма заявки (без авторизации)

**Что сделано:** Добавлен эндпоинт `POST /api/orders/public`.

**Изменение относительно первоначального плана:**
- В отчёте форма заявки доступна только авторизованному заказчику.
- **Принято решение:** форма на странице «Контакты» работает без авторизации.
- **Причина:** основная цель публичного сайта — привлечение новых клиентов, которые ещё не имеют аккаунта. Требование входа перед отправкой заявки создаёт лишний барьер.

**Следствие:** заявки без аккаунта хранятся с `customer_login = NULL`, видны только исполнителю.

**Файл:** `backend/routes/orders.js`

## [2026-06-16] Регистрация заказчиков

**Что сделано:** Добавлена страница регистрации для заказчиков.

**Изменение относительно первоначального плана:**
- В отчёте пользователи описаны как заранее созданные (seed-данные).
- **Принято решение:** реализовать самостоятельную регистрацию для роли `customer`.
- **Причина:** реальное приложение должно позволять новым клиентам создавать аккаунты без участия администратора.

**Детали реализации:**
- Backend: `POST /api/register` — проверка уникальности логина, минимальная валидация (логин ≥ 3 символов, пароль ≥ 6 символов, совпадение паролей), автоматическая авторизация после регистрации
- Frontend: страница `/register` с формой, ссылка с `/login`, редирект в кабинет заказчика после регистрации

**Файлы:** `backend/routes/auth.js`, `frontend/src/pages/RegisterPage.jsx`

## [2026-06-16] Чат внутри каждой заявки

**Что сделано:** Добавлен отдельный чат между заказчиком и исполнителем на странице конкретной заявки.

**Изменение относительно предыдущей реализации:**
- Ранее комментарий исполнителя хранился в поле `executor_comment` заявки и обновлялся вместе со статусом.
- **Принято решение:** разделить обычные сообщения и комментарии к статусам.
- **Причина:** чат и история изменения статусов имеют разный смысл: сообщения используются для общения по заявке, а комментарий к статусу объясняет конкретное изменение состояния заявки.

**Детали реализации:**
- Новая таблица `order_messages` хранит сообщения с привязкой к `order_id`.
- У каждой заявки свой изолированный чат.
- Сообщения отправляются через отдельный эндпоинт `POST /api/orders/:id/messages`.
- Статус заявки меняется через отдельный эндпоинт `PATCH /api/orders/:id/status`.
- Комментарий к статусу сохраняется только в истории статусов и не попадает в чат.

**Файлы:** `backend/routes/orders.js`, `backend/migrations/init.sql`, `frontend/src/components/OrderChat.jsx`

## [2026-06-16] Непрочитанные сообщения в списке заявок

**Что сделано:** В списки заявок заказчика и исполнителя добавлен индикатор новых сообщений.

**Изменение относительно предыдущей реализации:**
- Ранее пользователь узнавал о новых сообщениях только после открытия конкретной заявки.
- **Принято решение:** хранить момент последнего просмотра заявки пользователем и считать непрочитанные сообщения на сервере.
- **Причина:** список заявок должен сразу показывать, где есть новые сообщения, и поднимать последние активные заявки наверх.

**Детали реализации:**
- Новая таблица `order_reads` хранит `last_read_at` для пары заявка + пользователь.
- При открытии страницы заявки текущие сообщения считаются прочитанными.
- В списках заявок возвращаются поля `unread_count` и `last_activity_at`.
- Сортировка списков выполняется по последней активности: создание заявки, сообщение в чате или изменение статуса.

**Файлы:** `backend/routes/orders.js`, `backend/migrations/init.sql`, `frontend/src/pages/customer/OrdersList.jsx`, `frontend/src/pages/executor/AllOrdersList.jsx`

## [2026-06-16] Ограничение чата для публичных заявок

**Что сделано:** Для заявок, оставленных без авторизации заказчика, чат с исполнителем недоступен.

**Причина:** у публичной заявки нет аккаунта заказчика, поэтому невозможно корректно показать заказчику личный кабинет, чат и состояние прочтения сообщений.

**Детали реализации:**
- Backend возвращает флаг `chat_available` для страницы заявки.
- Отправка сообщений в чат публичной заявки запрещена на сервере.
- В кабинете исполнителя вместо чата показывается пояснение, что нужно связаться с клиентом по контактам из заявки.
- В модальном окне заявки неавторизованный пользователь видит предупреждение о недоступности чата.

**Файлы:** `backend/routes/orders.js`, `frontend/src/components/OrderModal.jsx`, `frontend/src/pages/executor/OrderManage.jsx`
